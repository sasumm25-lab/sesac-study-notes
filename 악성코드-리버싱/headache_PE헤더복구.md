# headache (2 bytes) — 깨진 PE 헤더 복구 크랙미

실습파일 `headache(2 bytes).exe`. 시리얼을 찾는 게 아니라 **일부러 망가뜨린 PE 헤더 2바이트를 하드에디터로 고쳐서 정상 실행되게 만드는** 문제다. 이름의 "2 bytes"가 곧 힌트 — 딱 두 바이트만 고치면 된다. 앞서 정리한 [PE 포맷](00_교안_실습파일_매핑.md) 지식이 그대로 쓰인다.

## 증상

파일을 열면 디버거/도구가 PE로 인식하지 못한다.

```
$ file "headache(2 bytes).exe"
headache(2 bytes).exe: MS-DOS executable      ← PE32가 아니라 도스 실행파일로 잡힘
```

`pefile` 등으로 파싱해도 실패한다. 헤더가 깨졌다는 신호다.

## 원인 — 손상된 2바이트

### ① 오프셋 0x01 : MZ 시그니처

파일 맨 앞 2바이트는 `4D 5A`(`"MZ"`)여야 한다. 이게 `4D 5D`로 깨져 있어 시그니처 검증부터 실패한다.

```
0x00:  4D 5D ...   (깨짐, "M]")
       ↓ 수정
0x00:  4D 5A ...   ("MZ")
```

**수정: 오프셋 `0x01` 의 `5D` → `5A`** (1바이트)

### ② 오프셋 0x3C : e_lfanew

DOS 헤더의 `e_lfanew`(0x3C, 4바이트 LE)는 **진짜 PE 헤더가 파일 어디에 있는지 가리키는 포인터**다. 이게 `0xF9`(쓰레기 위치)를 가리켜 `PE\0\0` 시그니처를 못 찾는다.

```
0x3C:  F9 00 00 00   → 0xF9 를 가리킴 (그 자리엔 쓰레기)
       ↓ 수정
0x3C:  E8 00 00 00   → 0xE8 를 가리킴 (진짜 PE 헤더)
```

`PE\0\0`(`50 45 00 00`) 시그니처는 원래 **`0xE8`에 멀쩡히 살아있다.** 포인터만 그리로 돌려주면 된다.

**수정: 오프셋 `0x3C` 의 `F9` → `E8`** (1바이트)

### 정리표

| 오프셋 | 필드 | 깨진 값 | 고친 값 | 바이트 수 |
|---|---|---|---|---|
| `0x01` | MZ 시그니처('Z') | `5D` | `5A` | 1 |
| `0x3C` | e_lfanew (PE 헤더 위치) | `F9` | `E8` | 1 |
| | | | **합계** | **2바이트** |

> **개념 정리**: 1바이트 = 16진수 두 자리(00~FF). `F9`→`E8`은 두 자리가 다 바뀌어도 **한 주소의 값 하나**가 바뀐 것이라 여전히 1바이트다. 바이트는 자릿수가 아니라 **위치(offset) 단위**로 센다.

## 고친 뒤 — 정상 PE 확인

두 바이트를 고치면 `e_lfanew=0xE8`이 `0xE8`의 `PE\0\0`을 가리키고, 뒤따르는 구조가 전부 유효해진다.

```
file 결과: PE32 executable (console) Intel 80386
섹션: .text / .rdata / .data / .rsrc (4개)
EP=0x11A0 (.text), ImageBase=0x400000, Subsystem=3(콘솔)
Characteristics에 RELOCS_STRIPPED → ASLR 없음, 항상 0x400000 로드
```

## 실행 결과 — 플래그

정상 실행하면 MessageBox가 뜬다.

```
제목: practice:reversing.
본문: flag{COOL_TECH}
```

이 플래그는 **어디서 계산되는 게 아니라 `.rdata`에 유니코드(UTF-16LE) 평문으로 그냥 박혀 있다.** 헤더를 못 고치면 실행이 안 돼서 이 창이 안 뜰 뿐, 문자열 자체는 숨김이 없다.

### 문자열 저장 위치

| 문자열 | VA | 파일 오프셋 | 섹션 |
|---|---|---|---|
| `practice:reversing.` (제목) | `0x409278` | `0x7878` | .rdata |
| `flag{COOL_TECH}` (본문) | `0x4092A0` | `0x78A0` | .rdata |

> 이 주소는 `Nop(quiz2 - template).exe`의 MessageBox 뼈대(caption `0x409278`, text `0x4092A0`)와 **완전히 동일**하다. 같은 템플릿으로 만든 문제.

### 호출 코드 (0x401000)

```asm
0x401000  push 0            ; uType = MB_OK
0x401002  push 0x409278     ; lpCaption -> "practice:reversing."
0x401007  push 0x4092A0     ; lpText    -> "flag{COOL_TECH}"   [*]
0x40100C  push 0            ; hWnd = NULL
0x40100E  call [0x4080E4]   ; USER32!MessageBoxW
```

Win32 API는 인자를 **역순 push**하므로, 세 번째로 들어가는 `push 0x4092A0`이 본문(`lpText`)이다.

## x32dbg에서 플래그 위치 확인하기

**주의**: 문자열 주소(`0x4092A0`)에 일반 BP(F2)를 걸면 안 멈춘다. 그 자리는 실행되는 코드가 아니라 데이터라, "실행 시 멈춤"인 소프트웨어 BP는 반응하지 않는다. (실제로 F9 하면 그냥 지나가고 플래그 창만 뜬다.)

**방법 1 — 코드에 BP (쉬움)**
- CPU 창에서 `0x40100E`(call MessageBoxW) 또는 `0x401007`(push flag)에 F2 → F9.
- 멈추면 스택 창에서 인자 `0x004092A0` 확인.

**방법 2 — 문자열 읽는 순간 잡기 (하드웨어/메모리 BP)**
- 덤프 창 `Ctrl+G` → `0x4092A0`
- `flag{...}` 바이트 드래그 선택 → 우클릭 → **Breakpoint → Hardware, Access**
- F9 → 그 문자열을 읽는 코드에서 멈춤.

덤프에서 `0x4092A0`을 보면 `66 00 6C 00 61 00 67 00 ...` 처럼 글자마다 `00`이 끼어 있다(UTF-16 특징: `f·l·a·g`).

## 배운 것

- **PE 로딩의 첫 관문**: MZ 시그니처(`0x00`) → e_lfanew(`0x3C`) → PE 시그니처(`0xE8`). 이 체인 중 하나만 끊겨도 실행이 안 된다.
- **소프트웨어 BP vs 하드웨어 BP**: 코드 실행엔 F2(소프트웨어), 데이터 읽기/쓰기엔 하드웨어(Access) BP.
- 악성코드가 분석을 방해하려고 헤더를 일부러 손상시키는 기법(파일이 도스 실행파일처럼 보이게)의 축소판이다.
