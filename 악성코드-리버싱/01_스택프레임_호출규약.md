# 스택 프레임과 호출 규약 (Stack Frame / cdecl / stdcall)

교안 p50~56. 실습파일 `Stack Frame.exe`, `cdecl.exe`, `stdcall.exe`를 x32dbg/디스어셈블로 뜯어 함수 호출의 내부 동작을 비교한다. 셋 다 `add(a, b)` 같은 단순 함수를 `main`에서 호출하는 구조라, 차이가 코드에 그대로 드러난다.

## 1. 스택 프레임 (Stack Frame.exe)

함수가 호출되면 `EBP`를 기준으로 **지역변수와 매개변수를 고정 오프셋으로 접근**하는 틀을 만든다. 이게 스택 프레임이다.

### 함수 프롤로그 / 에필로그

```asm
; --- add 함수 (0x401000) ---
0x401000  push ebp             ; 이전 프레임 포인터 저장
0x401001  mov  ebp, esp        ; 현재 스택 top을 프레임 기준으로
0x401003  sub  esp, 8          ; 지역변수 공간 8바이트 확보  ← 프롤로그
0x401006  mov  eax, [ebp+8]    ; 첫 번째 인자 a
0x401009  mov  [ebp-8], eax    ; 지역변수에 복사
0x40100C  mov  ecx, [ebp+0xC]  ; 두 번째 인자 b
0x40100F  mov  [ebp-4], ecx
0x401012  mov  eax, [ebp-8]
0x401015  add  eax, [ebp-4]    ; a + b → 반환값 EAX
0x401018  mov  esp, ebp        ; 스택 정리
0x40101A  pop  ebp             ; 이전 프레임 복원        ← 에필로그
0x40101B  ret
```

### EBP 기준 오프셋 지도

프롤로그 직후 스택은 이렇게 배치된다 (주소가 높은 쪽이 위):

```
[ebp+0xC]  두 번째 인자 (b)
[ebp+0x8]  첫 번째 인자 (a)
[ebp+0x4]  복귀 주소 (call이 push한 것)
[ebp+0x0]  저장된 이전 EBP   ← EBP가 여기를 가리킴
[ebp-0x4]  지역변수 1
[ebp-0x8]  지역변수 2        ← ESP가 여기 (sub esp,8)
```

- **매개변수는 `[ebp+8]`부터 양의 오프셋**, **지역변수는 `[ebp-4]`부터 음의 오프셋**. 이 규칙만 기억하면 낯선 함수도 인자/지역변수를 바로 구분할 수 있다.
- x32dbg에서 함수 진입 후 `F7`로 프롤로그를 밟으며 우측 스택 창을 보면 위 그림이 그대로 그려진다.

### main에서의 호출

```asm
0x401026  mov  [ebp-4], 1      ; a = 1
0x40102D  mov  [ebp-8], 2      ; b = 2
0x401034  mov  eax, [ebp-8]
0x401037  push eax             ; 인자 b 를 스택에 (역순 push)
0x401038  mov  ecx, [ebp-4]
0x40103B  push ecx             ; 인자 a
0x40103C  call 0x401000        ; add(1, 2) 호출 → 복귀주소 push
0x401041  add  esp, 8          ; 인자 2개(8바이트) 정리 ← 호출자가 치움
```

**인자는 오른쪽부터(역순) push**되고, `call`이 복귀 주소를 push한다는 점이 핵심이다.

## 2. cdecl vs stdcall — 누가 스택을 치우는가

`cdecl.exe`와 `stdcall.exe`는 사실상 같은 프로그램을 **호출 규약만 바꿔 컴파일**한 것이다. 두 바이너리를 같은 위치에서 비교하면 차이가 딱 두 군데다.

### add 함수의 끝 (callee 쪽)

```asm
; cdecl.exe                      ; stdcall.exe
0x401009  pop ebp                0x401009  pop ebp
0x40100A  ret                    0x40100A  ret 8      ; ← 인자 8바이트를 함수가 직접 정리
```

### 호출 직후 (caller 쪽, main)

```asm
; cdecl.exe                      ; stdcall.exe
0x401017  call 0x401000          0x401017  call 0x401000
0x40101C  add  esp, 8   ← 호출자   0x40101C  pop ebp    ; add esp 없음! 이미 함수가 치움
0x40101F  pop  ebp        가 정리
```

### 정리

| | **cdecl** | **stdcall** |
|---|---|---|
| 스택 정리 주체 | **호출자(caller)** | **피호출자(callee)** |
| 함수 끝 | `ret` | `ret 8` (인자 크기만큼) |
| 호출 직후 | `add esp, 8` 있음 | 없음 |
| 가변 인자(`printf`) | 가능 | 불가 |
| 대표 사용처 | C 표준 함수 | **Win32 API 전부** |

**리버싱 관점의 실전 의미**

- 함수 끝이 `ret N` (N≠0)이면 **stdcall**이고, N을 4로 나누면 인자 개수를 바로 알 수 있다. `ret 8` → 인자 2개.
- 함수 끝이 그냥 `ret`인데 호출부마다 `add esp, N`이 따라오면 **cdecl**이다.
- Win32 API는 전부 stdcall이라, IAT 호출 뒤에 `add esp`가 없는 게 정상이다. 이 감각이 있으면 디스어셈블리에서 함수 경계와 인자 수를 훨씬 빨리 읽는다.

x32dbg에서 확인: `add` 함수 마지막 줄(`ret` 또는 `ret 8`)과, `call` 바로 다음 줄에 `add esp,8`이 있는지 두 지점만 비교하면 규약이 판별된다.
