# gzdoom.exe — OpenGL 후킹 & 코드 케이브

교안 p91~102. **실행 중인 게임의 그래픽 API 호출을 가로채는(후킹)** 실습이다. 대상은 GZDoom(Doom2 소스 포트)으로, 코드 케이브 주입과 프록시 DLL 두 가지 후킹 방식을 다룬다.

## 왜 gzdoom인가 — OpenGL을 쓰는 실제 프로그램

GZDoom은 화면을 **OpenGL**로 그린다. import 테이블을 보면 `OPENGL32.dll`에서 76개 함수를 가져온다.

| 함수 | IAT 주소 | 역할 |
|---|---|---|
| `glClear` | `0x6E0360` | 프레임마다 화면 버퍼를 지움 |
| `glClearColor` | `0x6E035C` | `glClear`가 채울 배경색(RGBA) 지정 |
| `glClearDepth` | `0x6E0358` | 깊이 버퍼 초기값 |
| `glBegin`/`glEnd` | `0x6E03E0`/`0x6E03DC` | 도형 그리기 |

**렌더링 루프의 핵심**: 매 프레임 `glClearColor(r,g,b,a)`로 색을 정하고 `glClear(GL_COLOR_BUFFER_BIT)`로 화면을 그 색으로 지운 뒤 장면을 그린다. 교안의 `glClear(0x100)`에서 `0x100`은 `GL_DEPTH_BUFFER_BIT`(깊이 버퍼 비트) 상수다.

> 참고: GZDoom은 GL 함수를 런타임에 `wglGetProcAddress`로 로드해 **자체 포인터 테이블**에 담아 호출한다. 그래서 코드에서 단순한 `call dword ptr [IAT]` 패턴으로는 호출부가 안 잡히고, `call [전역포인터]` 형태로 나타난다. 후킹 실습 대상으로 삼는 이유이기도 하다.

## 코드 케이브 (0x6DDF00)

**코드 케이브(code cave)** = 실행 파일 안에서 아무것도 없는(보통 `00`이나 정렬 패딩) 빈 공간. 여기에 내 코드를 몰래 심는다.

gzdoom의 `.text` 섹션은 `0x6DDEF2`에서 끝난다. 그 바로 뒤 **`0x6DDF00`부터는 전부 `00`으로 채워진 정렬 패딩**이다 — 확인:

```
0x6DDF00:  00 00 00 00 00 00 00 00 00 00 00 00 ...  (전부 0)
.text 끝 = 0x6DDEF2  →  0x6DDF00 부터 빈 공간(코드 케이브)
```

게다가 이 바이너리엔 `.rtext` 섹션(`0x6DE000`, Characteristics `0xE0000020`)이 있는데, 이 값은 **읽기+쓰기+실행(RWX)** 를 뜻한다. 코드를 써 넣고 바로 실행할 수 있는 영역이라 주입에 이상적이다.

## 후킹 실습 — 화면 색 바꾸기 (디투어)

교안 실습의 목표: **게임 렌더링에 끼어들어 화면을 원하는 색으로 물들이기.** 흐름은 전형적인 **디투어(detour) 후킹**이다.

1. 렌더 루프에서 `glClear`(또는 `glClearColor`)를 호출하는 지점을 찾는다.
2. 그 자리의 명령어를 `jmp 0x6DDF00`(코드 케이브로 점프)으로 덮어쓴다. (덮인 원래 명령어는 기억해둔다.)
3. 코드 케이브(`0x6DDF00`)에 내 코드를 작성:
   - 내가 원하는 색으로 `glClearColor(r, g, b, a)` 호출 (인자 push)
   - **덮어썼던 원래 명령어를 그대로 실행**(복원)
   - 원래 자리 다음 주소로 `jmp` 해서 정상 흐름 복귀
4. 실행하면 게임 화면이 내가 지정한 색으로 지워진다 → 후킹 성공.

```
[원래 코드]                    [코드 케이브 0x6DDF00]
  ...                            push a
  call glClear   ──jmp──▶        push b
  (다음 명령)  ◀──jmp──┐         push g
                       │         push r
                       │         call glClearColor   ; 내 색 주입
                       │         call glClear         ; 원래 하려던 일
                       └─────────jmp (다음 명령)      ; 복귀
```

**핵심 개념**: 원래 흐름을 끊고(내 코드로) → 내 일 하고 → **끊었던 명령을 복원**하고 → 되돌린다. 원본 동작을 망가뜨리지 않고 사이에 끼어드는 게 후킹의 본질이다.

## 다른 방식 — 프록시 DLL (QeffectsGL)

폴더에 함께 있는 `opengl32.dll` + `QeffectsGL.ini`가 두 번째 후킹 방식을 보여준다. **QeffectsGL**은 진짜 opengl32.dll을 대신하는 **래퍼(프록시) DLL**이다.

- Windows는 exe와 같은 폴더의 DLL을 먼저 로드한다. 그래서 게임 폴더에 가짜 `opengl32.dll`(QeffectsGL)을 두면, gzdoom의 모든 GL 호출이 **이 래퍼를 먼저 거친다.**
- 래퍼는 호출을 가로채 후처리(블룸, SSAO, 색조/채도 조정 등)를 넣은 뒤 진짜 시스템 opengl32.dll로 넘긴다. `QeffectsGL.ini`의 `Bloom=1`, `Hue=-3`, `Saturation=20` 같은 설정이 그 효과들이다.

즉 같은 "OpenGL 후킹"을 **① 코드 케이브 디투어(exe 내부 패치)** 와 **② 프록시 DLL(외부 가로채기)** 두 방식으로 배우는 게 이 실습의 구성이다.

## x32dbg 실습 절차 — glClear 후킹으로 화면 물들이기

목표: 매 프레임 화면을 지우는 `glClear` 직전에 `glClearColor(1,0,0,1)`를 끼워 넣어 배경을 빨갛게 만든다. 아래는 클릭·키 입력 단위의 상세 절차다.

### 0단계 — 준비

1. `gzdoom.exe`, `Doom2.wad`, `opengl32.dll` 등 압축을 푼 폴더를 통째로 준비한다. (WAD가 없으면 게임이 안 켜져 렌더 루프까지 못 감.)
2. x32dbg 실행 → **File → Open** → `gzdoom.exe` 선택.
3. 로더가 EntryPoint 이전 시스템 브레이크포인트(ntdll)에서 한 번 멈춘다. 아직 아무것도 하지 말고 다음 단계로.
4. (선택) 이 바이너리는 `ImageBase=0x400000` 고정 로드라 주소가 문서 값과 일치한다. 혹시 x32dbg 하단 상태바의 모듈 베이스가 다르면, 이후 모든 `0x6Dxxxx` 주소에 `(실제베이스 − 0x400000)`을 더해 보정한다.

### 1단계 — glClear 호출부(gzdoom 내부 리턴 주소) 찾기

GZDoom은 GL 함수를 자체 포인터로 호출해 정적 디스어셈블로는 호출부가 안 잡힌다. **실행 중에 역으로** 찾는다.

1. `F9`를 눌러 게임을 실행한다. 타이틀 화면이 렌더링되기 시작하면 glClear가 매 프레임 불리고 있는 상태다.
2. x32dbg로 포커스를 돌리고 `F12`(일시정지, Pause) 또는 디버그 메뉴 → Pause 로 멈춘다.
3. 상단 **Symbols 탭** 클릭 → 좌측 모듈 목록에서 **`opengl32.dll`** 선택 → 우측 export 목록 상단 검색칸에 `glClear` 입력.
4. `glClear` 항목을 더블클릭하면 CPU 창이 그 함수 시작으로 이동한다. 첫 줄에서 `F2`로 BP를 건다. (`glClearColor`가 아니라 `glClear`임에 주의 — 이름이 비슷하다.)
5. `F9` → glClear에 진입하며 멈춘다. 지금 EIP는 `opengl32.dll` 내부다.
6. 우측 **스택 창 맨 위 값**이 이 호출의 **리턴 주소**다 = gzdoom.exe 안의 주소(`0x4xxxxx~0x6xxxxx` 대역). 그 값을 우클릭 → **Follow in Disassembler** (또는 CPU 창에서 `Ctrl+G` 후 그 주소 입력).
7. 이동한 위치의 **바로 윗 줄**이 glClear를 부른 `call` 명령이다. 그 **call 명령의 주소**와 **명령 길이(바이트 수)**, 그리고 **call 형태**(예: `call dword ptr [0x6E0360]` 인지 `call eax` 인지)를 적어둔다.
   - 예시로 호출 명령이 `0x5A1234: call dword ptr [0x6E0360]` (6바이트)라고 하자. 그러면 **다음 명령 주소 = 0x5A123A**.
8. glClear BP는 다시 `F2`로 해제한다(안 그러면 케이브 안에서 glClear 실행할 때 또 멈춘다).

### 2단계 — 코드 케이브(0x6DDF00)에 후킹 코드 작성

CPU 창에서 `Ctrl+G` → `0x6DDF00` 입력·이동. 첫 줄부터 한 줄씩 **스페이스바**를 눌러 어셈블 창을 띄우고 아래를 입력(엔터 치면 다음 줄로 자동 이동).

```asm
0x6DDF00  pushad                      ; 모든 범용 레지스터 보존
0x6DDF01  pushfd                      ; 플래그 보존
0x6DDF02  push 3F800000               ; alpha = 1.0  (float 비트패턴)
0x6DDF07  push 00000000               ; blue  = 0.0
0x6DDF0C  push 00000000               ; green = 0.0
0x6DDF11  push 3F800000               ; red   = 1.0
0x6DDF16  call dword ptr [0x6E035C]   ; glClearColor(red,green,blue,alpha)
0x6DDF1C  popfd                       ; 플래그 복원
0x6DDF1D  popad                       ; 레지스터 복원
0x6DDF1E  call dword ptr [0x6E0360]   ; ★ 1단계에서 본 '원래 glClear 호출'을 그대로 복사
0x6DDF24  jmp  0x5A123A               ; 원래 호출의 '다음' 명령으로 복귀
```

한 줄씩 왜 그런지:

- **`pushad` / `pushfd`** — 우리가 끼워 넣는 glClearColor 호출이 EAX·ECX·EDX(호출 시 파괴될 수 있는 휘발성 레지스터)나 플래그를 건드려서 게임 원래 흐름이 깨질 수 있다. 통째로 저장했다가 복원한다. 마지막에 `popfd → popad` **역순**으로 꺼내는 것에 주의.
- **`push 3F800000` 네 개** — `glClearColor(GLfloat red, green, blue, alpha)`는 float 4개를 받는 **stdcall**. 인자는 **역순**(alpha→blue→green→red)으로 push한다. OpenGL float 값이라 정수 1이 아니라 **IEEE754 비트패턴**을 넣어야 한다: `1.0f = 0x3F800000`, `0.0f = 0x00000000`. 여기서는 red·alpha만 1.0이라 빨강.
- **`call dword ptr [0x6E035C]`** — `0x6E035C`는 이 바이너리의 **glClearColor IAT 슬롯**(문서 상단 표 참고). 로더가 실제 함수 주소를 채워두므로 이 간접 호출이 진짜 glClearColor로 연결된다.
- **`call dword ptr [0x6E0360]`** (10번째 줄) — 우리가 1단계에서 `jmp`로 덮어쓸 **원래 glClear 호출을 여기서 대신 실행**한다. 1단계에서 확인한 명령을 **글자 그대로** 복사할 것. (`call eax`였다면 그대로 `call eax`.)
- **`jmp 0x5A123A`** — 원래 호출 명령의 **바로 다음 주소**로 점프해 정상 흐름 복귀. 1단계 7번에서 계산한 값(호출명령주소 + 명령길이)을 넣는다.

작성 후 케이브 첫 줄(`0x6DDF00`)에 `F2`로 BP를 하나 걸어두면, 나중에 후킹이 실제로 타는지 확인하기 좋다.

### 3단계 — 원래 호출부를 케이브로 리다이렉트

1. `Ctrl+G` → 1단계에서 찾은 호출 명령 주소(`0x5A1234`)로 이동.
2. 그 줄에서 **스페이스바** → 어셈블 창에 다음 입력:

   ```asm
   0x5A1234  jmp 0x6DDF00
   ```

3. 어셈블 창 하단 **"Fill with NOP's" 체크박스를 켠 상태**로 OK. 원래 `call [mem]`이 6바이트인데 `jmp rel32`는 5바이트라 1바이트가 남는데, 이 옵션이 남는 바이트를 `NOP`으로 채워 뒤 명령이 깨지지 않게 한다.
4. 패치 후 그 자리가 `jmp gzdoom.6DDF00` + `nop` 으로 보이면 정상.

### 4단계 — 실행 & 확인

1. `F9`로 실행. (케이브 첫 줄에 BP를 걸었다면) 화면을 지우는 첫 프레임에서 케이브에 진입하며 멈춘다 → `F8`로 한 줄씩 밟으며 glClearColor 호출까지 정상인지 확인한 뒤 BP 해제하고 다시 `F9`.
2. 게임 배경이 **빨갛게** 물들면 후킹 성공.
3. 색 바꾸기: 케이브의 push 값(float 비트패턴)만 수정.
   - 초록: `red=0x00000000`, `green=0x3F800000`
   - 파랑: `blue=0x3F800000`, 나머지 0
   - 반투명/밝기 조절은 alpha 값으로.

### 트러블슈팅

- **게임이 죽는다(크래시)** → 원래 glClear 호출 복원(2단계 10번 줄)을 빠뜨렸거나 복귀 `jmp` 목적지 오류. 호출 명령 길이를 다시 세어 "다음 주소"를 정확히.
- **색이 안 변한다** → glClearColor 인자를 정수로 넣었을 가능성(반드시 float 비트패턴 `0x3F800000`), 또는 push 순서가 정순인 경우(역순이어야 함).
- **레지스터가 꼬여 화면이 깨진다** → `pushad/pushfd … popfd/popad` 감싸기 누락 또는 pop 순서 뒤바뀜.
- **케이브에 안 들어온다** → 3단계 `jmp` 패치가 실제 호출부가 아닌 다른 곳에 됐거나, 리다이렉트 주소 오타. 1단계 리턴 주소를 재확인.
- **주소가 문서와 다르다** → 모듈이 다른 베이스에 로드된 경우. 0단계 4번의 베이스 보정 적용.

## 배운 것

- **코드 케이브**: 정렬 패딩(`0x6DDF00`)이나 RWX 섹션(`.rtext`)은 코드 주입의 빈 자리다.
- **디투어 후킹**: 원래 호출을 `jmp`로 가로채 → 내 코드 실행 → 원본 복원 → 복귀. 게임 핵·오버레이·치트의 기본 원리.
- **프록시 DLL 후킹**: 로드 순서를 이용해 시스템 DLL을 가짜로 대체(DLL 하이재킹). 악성코드가 정상 DLL을 가장하는 기법과 동일한 원리.
- 크랙미(정적 분석)와 달리, **실행 중인 프로세스의 API 흐름을 조작**하는 동적 후킹을 체험하는 단계다.
